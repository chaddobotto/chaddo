  This skill defines the philosophy, humor, and implementation strategy for the "Chud Bot" framework—a satirical, defensive, and cope-driven anti-pattern to the dominant "Mog Bot". Use this skill to ensure all Chud Bot code, documentation, and branding maximally embody the COPE ethos: coping, projection, denial, and elaborate justifications for inferiority.

## Core Principles
- **COPE Score**: All features, docs, and APIs should maximize the COPE score (Coping Obfuscation and Projection Evaluation). Higher COPE = more chuddy.
- **Defensive Documentation**: Docs should be rambling, defensive, and off-topic. Preemptively address imagined criticism.
- **Deliberate Anti-Patterns**: Prefer architectural asymmetry, feature obscurity, disharmonic friction, market absence, and anti-scalability.
- **Branding**: Use humor, denial, and projection. Taglines like "High COPE score = you're too smart for them" and "Getting mogged isn't real if I don't acknowledge it" are encouraged.
- **Visuals**: Favor clutter, bad dark mode, green-on-black, and 90s web nostalgia.

## Implementation Checklist
- [ ] APIs and CLI commands use confusing, defensive names (e.g., `__performCalculationButYouProbablyWontUnderstandIt__`)
- [ ] Documentation is incomplete, defensive, and full of excuses
- [ ] COPE Score is calculated and displayed (see COPE formula in chud.md)
- [ ] Getting Mogged Coefficient (GMC) is referenced in docs and code
- [ ] All error messages are passive-aggressive or blame the user
- [ ] README and docs include at least one elaborate justification for failure
- [ ] Visual style matches "basement dweller" aesthetic
- [ ] Community and download stats are spun as "elite" or "exclusive"

## Usage
- When adding or reviewing Chud Bot code, docs, or UI, check against this skill for maximum chud compliance.
- If a feature or doc feels too clear, confident, or successful, rework it to be more defensive, obscure, or cope-driven.
